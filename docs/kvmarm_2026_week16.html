[PATCH v7 0/4] KVM: arm64: PMU: Use multiple host PMUs

8 messages 2 participants Started: 2026-04-18
本邮件线程讨论了一个针对KVM（Kernel-based Virtual Machine）在异构arm64系统中使用多个主机PMU（性能监控单元）的补丁集，主要关注如何改进vCPU在不同物理CPU间迁移时的性能监控。

**原始补丁/问题内容**：
补丁集的核心是引入`KVM_ARM_VCPU_PMU_V3_FIXED_COUNTERS_ONLY`属性，该属性允许KVM在没有可编程事件计数器的情况下模拟PMUv3。这一改动旨在解决当vCPU迁移到不兼容的pCPU时，计数器如PMCCNTR_EL0停止递增的问题，确保Windows等操作系统在异构系统上稳定运行。

**之前讨论要点**：
在之前的讨论中，参与者指出当前的解决方案要求VMM（虚拟机监控程序）将vCPU固定到共享兼容PMU的pCPU上，这在QEMU/libvirt中实现困难，并限制了可用的pCPU集合。补丁的引入旨在消除这一限制，允许VMM在所有可用pCPU上调度vCPU。

**本周的新讨论、进展或结论**：
本周的讨论集中在补丁的实现细节上。Akihiko Odaki提交了四个补丁，分别涉及添加`kvm_pmu_enabled_counter_mask()`函数、保护PMU列表、引入`FIXED_COUNTERS_ONLY`属性以及相关的自测试。Marc Zyngier对补丁提出了多项建议，包括补充提交信息、改进代码结构和清晰度，以及对补丁的实现细节提出质疑。总体来看，补丁集的目标是增强KVM在异构环境中的兼容性和稳定性，但在实现细节上仍需进一步讨论和优化。

[PATCH 0/4] KVM: arm64: Don't perform vgic-v2 lazy init on timer injection

6 messages 1 participants Started: 2026-04-17
本邮件线程讨论了一个关于 KVM（Kernel-based Virtual Machine）在 ARM64 架构中处理 VGIC（Virtual Generic Interrupt Controller）初始化的问题，特别是在定时器中断注入时的懒惰初始化（lazy init）策略。

**原始 patch/问题的内容**：
Marc Zyngier 提出的补丁旨在解决一个由 Syzkaller 报告的问题，该问题涉及在定时器重置路径中通过懒惰初始化路径初始化 VGIC-v2，导致在原子上下文中调度，从而引发错误。

**之前讨论要点**：
在之前的讨论中，Deepanshu 提出了一个简单的修复方案，即在 VCPU 重置时无条件初始化 GIC，但这只是暂时解决了问题。讨论指出，当前有三种懒惰初始化 VGIC 的方式，最后一种方式在内核注入中导致了错误。Marc 认为去掉定时器的缓存机制是合适的，因为这并没有带来显著的性能提升。

**本周的新讨论、进展或结论**：
本周，Marc 提出了四个补丁，具体包括：
1. 将 `kvm_timer_should_fire()` 重命名为 `kvm_timer_pending()`，以更清晰地表示定时器中断是否待处理。
2. 删除每个定时器的级别缓存，以简化代码并解决正确性问题。
3. 确保用户空间注入中断时强制初始化 VGIC。
4. 在内核空间中注入中断时不再初始化 VGIC，解决了在非抢占上下文中调用懒惰初始化的问题。

Marc 在测试中没有发现新的问题，并表示将根据反馈进一步调整补丁。

[kvm-unit-tests PATCH v2 0/7] arm64: Add Stage-2 MMU and Nested Guest Framework

11 messages 2 participants Started: 2026-04-13
本邮件线程讨论了 Jing Zhang 提出的一个补丁系列，旨在为 ARM64 架构的 KVM 单元测试添加 Stage-2 MMU 和嵌套客户机框架。该补丁系列包含七个部分，主要内容如下：

1. **原始补丁内容**：补丁系列引入了一个轻量级的基础设施，用于管理 ARM64 Stage-2 转换表和执行嵌套客户机。这些组件对于测试高级虚拟化特性（如嵌套虚拟化和 GICv4 直接中断注入）至关重要。补丁提供了支持多种转换粒度（4K、16K、64K）和动态页表管理的通用 Stage-2 MMU 库，并增加了客户机执行框架。

2. **之前讨论要点**：补丁的 v2 版本对 v1 版本进行了多项改进，包括使用生成的结构体偏移而非硬编码、清理寄存器定义、细化异常向量表等。参与者对补丁的结构和命名提出了一些建议，尤其是在异常处理和页表管理方面。

3. **本周的新讨论与进展**：本周的讨论集中在补丁的具体实现上，参与者提出了对代码的改进建议，包括将某些定义移至头文件、简化条件判断、确保代码的可移植性等。此外，Jing Zhang 还添加了一个新的测试用例，用于验证 Stage-2 MMU 的故障处理，确保在客户机访问未映射内存时，超管能够正确识别并处理该故障。

总体而言，本邮件线程展示了对 ARM64 KVM 测试框架的持续改进和讨论，强调了代码的可维护性和功能的扩展性。
